# Timeline Hover Tooltip Design

**Date**: 2026-04-07
**Status**: Draft
**Audience**: Humpback Timeline Viewer contributors

## 1. Summary

This design updates the active timeline viewer's hover affordances so overlay
details are readable and positioned where the user expects them. The current
canvas-backed viewport already exposes hover details for detections, but the
tooltip is anchored near the top of the track and then shifted upward, which
causes it to render outside the clipped viewport. Vocalizations currently have
no corresponding hover detail in the active canvas path.

This pass adds a shared hover-card model for both overlay types:

1. detection hover cards render near the actual pointer location inside the
   timeline track
2. vocalization hover cards appear for hovered vocalization windows
3. both cards use a consistent confidence presentation built around the label
   `Confidence:`

## 2. Context

The active viewer renders detections and vocalizations inside
[TimelineViewport](/Users/michael/development/humpback-annotation-app/frontend/src/components/timeline/TimelineViewport.tsx)
using canvas geometry generated by
[timeline-overlay-geometry.ts](/Users/michael/development/humpback-annotation-app/frontend/src/lib/timeline-overlay-geometry.ts).

Current behavior:

- detections are hit-tested through `findDetectionRectAtPoint(...)`
- hovered detection state stores only a detection id
- the detection tooltip is placed at a fixed `top: 12px`
- `.timeline-tooltip` applies `transform: translateY(-110%)`
- `.timeline-track` has `overflow: hidden`

That combination makes the current detection tooltip easy to clip outside the
viewport. The issue is structural, not just visual polish.

The timeline manifest already contains the needed hover data:

- detections expose `label`, `avg_confidence`, and `peak_confidence`
- vocalization labels expose `type`, `confidence`, and shared time windows
- visible vocalization windows are already grouped by `start_utc` and `end_utc`

No export-contract change is needed.

## 3. Goals

This design should:

1. keep hover cards visible inside the timeline track
2. position the card near the user’s hovered location instead of a fixed track
   corner
3. add hover detail for vocalization windows in the active canvas viewport
4. present vocalization window detail as all vocal types in the window plus an
   average confidence
5. update detection tooltip copy so confidence uses the same `Confidence:`
   label pattern
6. preserve the current manifest schema, playback behavior, and overlay mode
   semantics

## 4. Non-Goals

Out of scope for this change:

- changing detection or vocalization data structures in `manifest.json`
- allowing overlay editing
- changing overlay-mode exclusivity
- introducing click-to-pin, sticky inspectors, or mobile long-press behavior
- redesigning the whole timeline shell

## 5. Approaches Considered

### Approach A: Shared pointer-anchored hover cards in `TimelineViewport`

Track the pointer position relative to the timeline track, hit-test whichever
overlay is active, and render one DOM hover card whose content changes based on
the hovered overlay entity.

Pros:

- directly fixes the clipping issue
- keeps hover logic centralized in the active canvas viewport
- works for both detections and vocalizations
- avoids adding manifest or router complexity

Cons:

- requires new hover state and geometry helpers
- needs explicit tooltip clamping logic inside the track bounds

### Approach B: Per-overlay DOM hotspots layered over the canvas

Create transparent DOM elements for each detection and vocalization window and
use ordinary DOM hover behavior plus CSS-positioned tooltips.

Pros:

- simpler browser-native hover semantics
- less custom hit-testing code

Cons:

- duplicates geometry that is already computed for canvas rendering
- risks DOM/canvas drift at narrow zoom levels
- adds more layout work for dense windows

### Approach C: Minimal CSS fix for detections plus browser `title` for
vocalizations

Only move the detection tooltip down into the track and expose vocalization
detail through native browser tooltips or very small ad hoc handlers.

Pros:

- smallest diff
- fastest to ship

Cons:

- does not create a coherent hover system
- produces inconsistent detection vs vocalization behavior
- limits future overlay detail improvements

## 6. Decision

Choose **Approach A**.

The existing viewer already centralizes overlay rendering and hit-testing inside
the canvas viewport. Extending that model gives us the smallest coherent change:
one hover state shape, one tooltip component, one positioning strategy, and no
new export-contract surface.

## 7. Proposed Design

### 7.1 Shared Hover State

Replace the detection-only hovered id state with a shared hovered overlay model
that stores:

- overlay kind: `detection` or `vocalization`
- the hovered draw object or derived tooltip payload
- the pointer anchor position inside the track

This moves the tooltip from "find object, then anchor to object start x" to
"find object, then anchor to the actual hover location."

The track should continue clearing hover state during drag interactions and on
pointer leave.

### 7.2 Tooltip Positioning

The hover card should render inside `.timeline-track` and be clamped to the
track bounds with a small inset margin.

Behavior:

- default to appearing slightly to the right and below the pointer
- if the card would overflow the right edge, shift it left
- if the card would overflow the bottom edge, flip it upward
- keep the card fully inside the clipped track so `overflow: hidden` no longer
  hides it
- remove the fixed upward translation that currently pushes the card off-screen

This keeps the popup visually attached to the hover position while still
respecting the rounded, clipped viewport.

### 7.3 Detection Hover Content

Detection hover should keep the current detection label line and change the
metric copy to a unified confidence label.

Proposed content:

- title: detection label, or `Unlabeled`
- detail line: `Confidence: 74% avg, 91% peak`

This keeps the detection tooltip more informative than vocalizations while still
matching the requested confidence treatment.

### 7.4 Vocalization Hover Content

Add hover hit-testing for vocalization windows produced by
`buildVocalizationDrawWindows(...)`.

For a hovered vocalization window, the hover card should show:

- title: `Vocalizations`
- type line containing all vocal types in the hovered time window
- confidence line: `Confidence: 76% avg`

Detailed rules:

- derive the type list from the grouped window labels
- de-duplicate repeated type names while preserving first-seen order
- compute the average confidence across all labels in the hovered window
- round displayed percentages the same way detections do

At coarse zoom levels where label chips are hidden, the hover target should
still exist through the rendered vocalization window geometry so the user can
inspect the window.

### 7.5 Vocalization Hit Testing

Add a vocalization hit-test helper alongside detection hit testing.

The hit target should include the visible vocalization window region rather than
requiring the pointer to land on painted text only. That keeps hover usable:

- at `15m`, where indicator bars render but text labels do not
- in dense overlaps where the chip stack is compact
- when the pointer is near the rendered vocalization block instead of a precise
  glyph

Because overlay mode is exclusive in the current viewer, hit-testing can prefer
the active overlay type and does not need to resolve simultaneous detection and
vocalization conflicts.

### 7.6 Styling Direction

Keep the existing tooltip visual language, but tune it for the new anchored
placement:

- preserve the dark translucent panel treatment
- retain compact typography
- allow slightly wider content than the current 180px cap if needed for
  multiple vocal types
- keep pointer-events disabled so the tooltip never interrupts drag/seek input

## 9. Risks and Mitigations

### Risk: Tooltip placement still clips near track edges

Mitigation:

- clamp against measured track width and track height
- use a fixed inset margin instead of relying on CSS transforms alone

### Risk: Vocalization hover feels hard to trigger at dense or coarse zooms

Mitigation:

- hit-test against the window geometry, not just rendered text
- keep the hit target generous enough for slim indicators

### Risk: The tooltip becomes too wide when many vocal types share a window

Mitigation:

- allow line wrapping or a modestly wider max width
- preserve deduplicated type ordering instead of repeating duplicates
